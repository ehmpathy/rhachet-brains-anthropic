# vision: brain atom tools

---

> **✅ validated:** tools + structured output work together. anthropic docs confirm "if tools are specified, the model must support simultaneous use of tools and structured output." supported on sonnet 4.5, opus 4.5, and haiku 4.5.

## the outcome world

### day-in-the-life: before

```ts
// want to give claude access to a calculator tool? can't.
// atoms are inference-only, no tool use support.
// must use repl (heavy, sandboxed) or build custom loop.

const atom = genBrainAtom({ slug: 'claude/sonnet' });

// stuck with pure inference
const result = await atom.ask({
  prompt: 'what is 47 * 89?',
  schema: { output: z.object({ answer: z.number() }) },
});
// claude guesses: { answer: 4183 }
// wrong. no calculator to verify.
```

### day-in-the-life: after

```ts
import { genBrainPlugToolDeclaration, asBrainPlugToolDict } from 'rhachet';

// define tool with typed schemas
const weatherLookup = genBrainPlugToolDeclaration({
  slug: 'weather.lookup',
  name: 'Weather Lookup',
  description: 'get current weather for a city',
  schema: {
    input: z.object({ city: z.string() }),
    output: z.object({ temp: z.number(), conditions: z.string() }),
  },
  execute: async ({ invocation }) => {
    const weather = await fetchWeather(invocation.input.city);
    return { temp: weather.temp, conditions: weather.conditions };
  },
});

// build tool dict for O(1) lookup
const tools = [weatherLookup];
const toolDict = asBrainPlugToolDict(tools);

// initial prompt
let result = await brainAtom.ask({
  prompt: 'what is the current weather in austin?',
  schema: { output: z.object({ summary: z.string() }) },
  plugs: { tools },
});

// loop until output is defined
while (result.calls) {
  // execute each invocation in parallel
  const executions = await Promise.all(
    result.calls.tools.map(async (invocation) => {
      const tool = toolDict[invocation.slug];
      return tool.execute({ invocation }, context);
    }),
  );

  // continue episode with tool results
  result = await brainAtom.ask({
    on: { episode: result.episode },
    prompt: executions,  // BrainPlugToolExecution[]
    schema: { output: z.object({ summary: z.string() }) },
    plugs: { tools },
  });
}

// result.output is now defined
console.log(result.output.summary);
```

> **rhachet pattern:** `genBrainPlugToolDeclaration` creates reusable tools with type inference and automatic `BrainPlugToolExecution` wrapper. same tools work with both atoms (you call `execute`) and repls (repl calls `execute` internally).

### the "aha" moment

> "wait, i can plug any tool into an atom and claude will call it? and i control execution? and it works with structured output too?"

the value clicks when you realize:
- atoms stay lightweight (no sandbox, no agent loop)
- you control tool execution (your code, your rules)
- tools + structured output work together
- episode continuation carries tool context forward

## user experience

### usecases

| goal | tools plugged | outcome |
|------|---------------|---------|
| verified math | calculator | brain calls tool, you compute, return verified answer |
| data lookup | database query | brain requests query, you execute, return results |
| api calls | http client | brain constructs request, you send, return response |
| code execution | sandbox runner | brain writes code, you run, return output |
| multi-step workflows | multiple tools | brain orchestrates, you execute each step |

### contract inputs (from rhachet)

```ts
// create reusable tool declarations via factory
const tool = genBrainPlugToolDeclaration({
  slug: 'my-tool',
  name: 'My Tool',
  description: 'does something useful',
  schema: {
    input: z.object({ ... }),
    output: z.object({ ... }),
  },
  execute: async ({ invocation }) => { ... },  // returns output, wrapper adds time + signal
});

// ask with tools — rhachet contract
await atom.ask({
  prompt: AsBrainPromptFor<TPlugs>;  // string | BrainPlugToolExecution[] when tools plugged
  schema: { output: z.Schema<TOutput> };
  plugs?: {
    tools?: BrainPlugToolDefinition[];  // ← enables tool use
  };
  role?: { briefs?: Artifact<GitFile>[] };
  on?: { episode?: BrainEpisode };  // ← for continuation
});

// asBrainPlugToolDict — O(1) lookup with per-slug type preservation
const toolDict = asBrainPlugToolDict([tool1, tool2] as const);
const tool = toolDict[invocation.slug];  // typed lookup
const execution = await tool.execute({ invocation }, context);

// BrainPlugToolExecution (from rhachet) — for tool result continuation
type BrainPlugToolExecution<TInput, TOutput> = {
  exid: string;           // correlation ID from invocation
  slug: string;           // which tool was executed
  input: TInput;          // what the tool was called with
  signal: 'success';
  output: TOutput;        // what the tool returned
  metrics: { cost: { time: IsoDuration } };
} | {
  exid: string;
  slug: string;
  input: TInput;
  signal: 'error:constraint' | 'error:malfunction';
  output: { error: Error };
  metrics: { cost: { time: IsoDuration } };
};
```

### contract outputs

```ts
// output when tools are plugged
interface BrainOutput<TOutput, 'atom', { tools: [...] }> {
  output: TOutput | null;        // null when brain defers to tools
  calls: {
    tools: BrainPlugToolInvocation[];  // tool calls to execute
  } | null;
  metrics: BrainOutputMetrics;
  episode: BrainEpisode;         // for continuation with tool results
  series: null;                  // atoms don't have series
}

// tool invocation shape
interface BrainPlugToolInvocation<TInput = unknown> {
  exid: string;    // correlation id for tool result
  slug: string;    // which tool was called
  input: TInput;   // parsed arguments
}
```

### timeline

1. **define tools** — use `genBrainPlugToolDeclaration` to create reusable tools with zod schemas
2. **create dict** — use `asBrainPlugToolDict` for O(1) lookup by slug
3. **ask with plugs** — pass tools via `plugs.tools`
4. **check calls** — if `result.calls`, brain wants to use tools
5. **execute tools** — lookup via `toolDict[slug]`, call `tool.execute({ invocation }, context)`
6. **continue episode** — call `atom.ask` with `BrainPlugToolExecution[]` as `prompt`
7. **receive output** — brain produces final `output` with tool context

### grain-based behavior

| grain | tool execution | `execute` on definition |
|-------|----------------|------------------------|
| atom  | outputs invocations → you execute → continue via `prompt: BrainPlugToolExecution[]` | optional (you call it) |
| repl  | executes tools internally via `tool.execute()` | required |

`genBrainPlugToolDeclaration` creates repl-grain tools. for atoms, reuse the same tools — call `execute` yourself.

### error classification

`genBrainPlugToolDeclaration` wraps `execute` with automatic error classification:

| error type | signal | meaning |
|------------|--------|---------|
| `BadRequestError` | `'error:constraint'` | caller violated constraints (4xx) |
| all other errors | `'error:malfunction'` | unexpected failure (5xx) |

this enables the brain to reason about failure modes and adapt.

## evaluation

### how well does it solve the goals?

| goal | solved? | notes |
|------|---------|-------|
| plug tools into atoms | ✅ | via `plugs.tools` |
| brain decides tool use | ✅ | claude chooses when to call |
| you control execution | ✅ | calls returned, you execute |
| works with output schema | ✅ | output is schema-validated |
| works with continuation | ✅ | episode carries tool context |
| type-safe tool inputs | ✅ | zod schema validation |

### pros

- **lightweight** — no sandbox, no agent loop, just inference + tool calls
- **controllable** — you execute tools, not the sdk
- **composable** — tools + schemas + briefs + continuation all work together
- **progressive** — no tools = simple output; with tools = nullable output + calls
- **type-safe** — zod schemas for both output and tool inputs

### cons

- **manual loop** — you handle the execute-and-continue loop (unlike repl)
- **parallel execution is caller's job** — claude can request multiple tools in one response; you execute and return all results
- **episode management** — must track and pass episode for continuation

### edgecases & pit of success

| edgecase | pit of success |
|----------|----------------|
| brain calls unknown tool | validate `call.slug` against your tool list |
| invalid tool input | zod schema fails, throw helpful error |
| tool execution fails | include error in tool result, brain adapts |
| brain doesn't use tools | `output` is set, `calls` is null — handle both |
| multiple tool calls | `calls.tools` is array, execute all |
| tool result too large | truncate or summarize before continuing |

## validated questions & implementation notes

### validated assumptions

1. ✅ **tools + structured output** — anthropic docs confirm they work together on sonnet 4.5, opus 4.5, haiku 4.5
2. ✅ **zod → json schema** — anthropic sdk provides utilities; rhachet uses zod schemas
3. ✅ **tool results via continuation** — implementation encodes in prompt/episode, emits `tool_result` blocks
4. ✅ **parallel tool calls** — claude can call multiple tools in single response; `calls.tools` array handles this

### implementation notes

1. **strict mode** — enable `strict: true` on tool input schemas by default. anthropic docs: "ensures claude's tool calls always match your schema exactly."

2. **parallel control** — `disable_parallel_tool_use=true` forces single tool use. implementation can expose this option if needed.

3. **cache behavior** — tools count toward cache prefix. hierarchy: `Tools → System → Messages`. changes to tools invalidate cache. minimum cacheable: 1,024 tokens (sonnet), 4,096 tokens (opus/haiku 4.5).

4. **haiku support** — haiku 4.5 supports standard tool calls. the known limitation (no episode continuation with structured outputs) is a separate concern from tool use itself.

## what is awkward?

### continuation for tool results

the caller must:
1. check if `calls` is set
2. execute each tool
3. build `BrainPlugToolExecution[]` with signal, output, metrics
4. call `atom.ask` again with `on.episode` and executions as `prompt`

this is more manual than repl's automatic loop. but that's the tradeoff: control vs convenience.

**rhachet prescribes the format:** `BrainPlugToolExecution` provides a typed contract for tool results with discriminated signals (`success`, `error:constraint`, `error:malfunction`). the supplier implementation translates this to provider-specific formats (anthropic `tool_result` blocks, openai tool messages, etc).

### nullable output type

when tools are plugged, `output` becomes `TOutput | null`. this forces callers to handle the null case even if they know the brain will answer directly.

**this is intentional** — it prevents accidental assumptions. but it may feel awkward for simple cases.

### schema conversion

`BrainPlugToolDefinition.schema.input` is a zod schema. anthropic needs json schema. the anthropic sdk has zod utilities for this conversion (validated assumption #2).
