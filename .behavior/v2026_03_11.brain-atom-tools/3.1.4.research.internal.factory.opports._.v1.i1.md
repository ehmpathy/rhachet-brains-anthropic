# research: factory opportunities

## the constraint

**the bottleneck:** API call latency in integration tests

each tool use test requires 2+ API calls minimum:
1. initial prompt → tool_use response (~1-2s)
2. continuation → final output (~1-2s)

**if you could fix one problem:** faster API responses

but this is external to our control. anthropic's API latency is what it is.

---

## flow and queues

**queues in verification:**
1. npm run test:integration queue — runs tests sequentially
2. anthropic API queue — rate limits apply

**what must you wait for:**
- API response latency (1-3s per call)
- haiku is fastest (~500ms latency), sonnet/opus slower

**work-in-progress limit:**
- tests run serially
- could parallelize but API rate limits would become the bottleneck

---

## waste

**handoffs:** none significant. tests run locally.

**repeated setup:**
- anthropic client instantiation (cached via genBrainAtom)
- tool definitions (can define once, reuse across tests)

**partial work:**
- none. each test is atomic.

**manual and error-prone:**
- none. all automated via jest.

---

## small batches

**verify incrementally:** ✅ yes

smallest verifiable unit:
1. single tool invocation test
2. single tool result continuation test

can get feedback before whole is done:
- run single test file: `npm run test:integration -- genBrainAtom.integration.test.ts`
- run single test case: `npm run test:integration -- -t "case5"`

---

## autonomy

**do you have all you need:** ✅ yes

- API key available
- test infra available
- rhachet types available
- no external teams to consult

**who must be consulted:** no one

## density

**% actual test vs setup/wait:**
- setup: ~5% (atom creation, tool definition)
- API wait: ~90% (latency)
- assertion: ~5% (fast, in-memory)

**where is time wasted:**
- API latency is inherent, not wasted
- no obvious reclaim opportunities

## systems view

**would this improvement help all future work:** marginal

API latency affects all integration tests. but:
- haiku is already fast (~500ms)
- useThen already shares calls
- test count is small

**not in local optimum trap.** current approach is reasonable.

## opportunity identified

**if one improvement:** reusable test tool definitions

create a set of standard test tools once:
```ts
// src/.test/tools/calculator.tool.ts
export const calculatorTool = genBrainPlugToolDeclaration({
  slug: 'test.calculator',
  name: 'Calculator',
  description: 'performs math operations',
  schema: {
    input: z.object({ a: z.number(), b: z.number(), op: z.enum(['+', '-', '*', '/']) }),
    output: z.object({ result: z.number() }),
  },
  execute: async ({ invocation }) => ({
    result: eval(`${invocation.input.a} ${invocation.input.op} ${invocation.input.b}`),
  }),
});
```

benefits:
- reuse across tests
- known deterministic behavior
- fast to execute locally
