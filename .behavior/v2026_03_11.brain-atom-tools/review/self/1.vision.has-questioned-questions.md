### question 1: structured output + tools compatibility

**question:** does `output_format` work alongside `tools`? or are they mutually exclusive?

**can answer via research?** yes — websearch

**answer:** they work together. anthropic docs state "if tools are specified, the model must support simultaneous use of tools and structured output." supported on claude sonnet 4.5, opus 4.5, and haiku 4.5.

**update:** the `output_format` parameter has moved to `output_config.format` in GA. the old beta header (`structured-outputs-2025-11-13`) remains supported in transition period.

**source:** [structured outputs docs](https://platform.claude.com/docs/en/build-with-claude/structured-outputs)

**verdict: ✅ answered — compatible on supported models**

### question 2: strict mode for tool input schemas

**question:** should we enable `strict: true` on tool input schemas?

**can answer via research?** yes — websearch

**answer:** yes. anthropic docs recommend: "add `strict: true` to your tool definitions to ensure claude's tool calls always match your schema exactly, to eliminate type mismatches or absent fields. perfect for production agents where invalid tool parameters would cause failures."

**recommendation:** enable `strict: true` by default in implementation

**source:** [implement tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/implement-tool-use)

**verdict: ✅ answered — enable strict mode**

### question 3: parallel tool calls

**question:** can claude request multiple tools at once? how do we surface that?

**can answer via research?** yes — websearch

**answer:** yes. claude can call multiple tools in parallel within a single response. all `tool_use` blocks are included in a single assistant message, and all correlated `tool_result` blocks must be provided in the subsequent user message.

**control:** `disable_parallel_tool_use=true` forces single tool use. available when `tool_choice.type` is `auto`, `any`, or `tool`.

**surface:** `result.calls.tools` is already an array in rhachet's `BrainOutput` contract — this naturally handles multiple tool calls.

**source:** [tool use overview](https://docs.claude.com/en/docs/agents-and-tools/tool-use/overview)

**verdict: ✅ answered — array contract already supports parallel**

### question 4: cache tokens for tool definitions

**question:** do tool definitions count toward cache? impact on cost?

**can answer via research?** yes — websearch

**answer:** yes. tools are part of the cache prefix. hierarchy: `Tools → System → Messages`. changes to tools invalidate the cache.

**cost:**
- cache writes: 25% more than base input tokens (5-minute TTL)
- cache reads: 10% of base input token cost
- minimum cacheable: 1,024 tokens (sonnet), 4,096 tokens (opus/haiku 4.5)

**impact:** static tool definitions benefit from cache. alter tools mid-conversation and cache invalidates.

**source:** [prompt cache docs](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)

**verdict: ✅ answered — tools cacheable, standard cost**

---

### question 5: haiku tool use support

**question:** does haiku support tools? (continuation doesn't work with structured output on haiku)

**can answer via research?** yes — websearch + extant brief

**answer:** yes, haiku 4.5 supports standard tool call. it even supports fine-grained tool stream.

**limitation:** haiku does NOT support `tool_reference` blocks (tool search) — an advanced feature.

**known issue (from our briefs):** haiku does not support episode continuation with structured outputs. this is separate from tool use support.

**source:** [haiku capabilities](https://designforonline.com/ai-models/anthropic-claude-haiku-4-5/)

**verdict: ✅ answered — haiku supports tools, continuation limit is separate**

### question 6: tool result continuation format (design question)

**question:** how should tool results be passed back?
- option A: encode in prompt string (implementation parses)
- option B: extend rhachet contract with explicit parameter
- option C: store in episode exchange structure

**can answer via research?** partially — need to check rhachet contract

**research:** checked rhachet `BrainEpisode` and `BrainExchange` types. `BrainExchange` has `input: string` and `output: string`. tool results would need encode step.

**analysis:**
- option A: works with current contract, implementation complexity in parse step
- option B: requires rhachet contract change — out of scope for this wish
- option C: exchanges are string-based, would need schema change

**answer:** option A is the pragmatic choice. implementation parses tool results from episode/prompt and emits proper `tool_result` blocks. this keeps rhachet contract unchanged.

**note:** the vision already acknowledges this: "implementation translates to anthropic tool_result format"

**verdict: ✅ answered — option A, implementation handles encode**
