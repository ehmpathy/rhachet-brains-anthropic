# self-review: has-questioned-assumptions

## hidden assumptions in the vision

---

### assumption 1: structured output works with tools

**what we assume:** `output_format` (structured outputs beta) works alongside `tools` parameter

**evidence?** none — the anthropic docs I read didn't explicitly confirm this works together

**what if opposite is true?** if they're mutually exclusive:
- tools would require raw text output (no schema)
- OR we'd need to drop tools support for structured output use case
- this would fundamentally change the vision

**did wisher say this?** no — wisher said "w/ and w/o output schema" which might acknowledge this uncertainty

**verdict: ⚠️ critical assumption — must spike before implementation**

---

### assumption 2: episode continuation works for tool results

**what we assume:** we can pass tool results via the same `on: { episode }` continuation mechanism

**evidence?** partial — anthropic docs show tool results as `tool_result` content blocks in messages array, which is different from our episode exchange model

**what if opposite is true?** tool result continuation might need:
- different message format than current episode.exchanges
- separate api parameter (not via prompt)
- stateful session on anthropic's side

**did wisher say this?** no — wisher just said test tool invocation

**counterexample:** current episode exchanges store `input` and `output` as strings. tool_result blocks are structured objects with `tool_use_id`.

**verdict: ⚠️ assumed — needs design alignment**

---

### assumption 3: zod-to-json-schema conversion works for tool inputs

**what we assume:** `zodToJsonSchema` or similar converts zod schemas correctly for anthropic's tool input_schema

**evidence?** anthropic sdk has zod utilities, but I haven't verified they work for tool definitions (vs just structured output)

**what if opposite is true?** we'd need custom schema conversion logic

**did wisher say this?** no — inferred from rhachet's use of zod for schemas

**verdict: 🟡 likely works, but should verify**

---

### assumption 4: atoms stay single-turn with tool calls

**what we assume:** after brain returns tool calls, caller executes tools and continues in a new atom.ask call

**evidence?** this is how the rhachet contract is designed (BrainOutput.calls)

**what if opposite is true?** anthropic's tool loop could be:
- automatic (server-side) — but that's server tools, not client tools
- streamed with multiple tool calls — would need different contract

**did wisher say this?** no — but aligns with atom's stateless nature

**exception:** what if brain wants to call multiple tools in sequence? would require multiple ask calls.

**verdict: ✅ reasonable assumption given atom semantics**

---

### assumption 5: haiku supports tools

**what we assume:** all models (haiku, sonnet, opus) support tool use

**evidence?** anthropic docs show tool use is generally available, but haiku already has limitations (no continuation with structured output)

**what if opposite is true?** haiku might not support tools, or might have limitations

**did wisher say this?** no

**verdict: 🟡 likely works, but should test per model**

---

### assumption 6: tool calls and output can coexist

**what we assume:** a single response can have BOTH tool_use blocks AND a final output

**evidence?** partial — anthropic docs show responses can have multiple content blocks

**what if opposite is true?** if tool calls always mean "no output yet":
- our `output: TOutput | null` design is correct
- but we should verify brain can't give partial output + tool calls

**did wisher say this?** no

**verdict: 🟡 design handles this, but should verify behavior**

---

### assumption 7: tool execution is caller's responsibility

**what we assume:** we return tool calls, caller executes, caller continues

**evidence?** this is the rhachet BrainPlugs contract design

**what if opposite is true?** an "auto-execute" mode would be convenient but:
- requires caller to provide executors upfront
- becomes closer to repl behavior
- violates atom's "you control" principle

**did wisher say this?** no — but consistent with wish's "tools are successfully invocable" focus

**verdict: ✅ deliberate design choice, well-reasoned**

---

### assumption 8: one tool definition format works for all providers

**what we assume:** BrainPlugToolDefinition shape (slug, description, schema.input) maps cleanly to anthropic

**evidence?** partial — anthropic uses (name, description, input_schema) which is similar

**map:**
- `slug` → `name` (1:1)
- `description` → `description` (1:1)
- `schema.input` → `input_schema` (zod → json schema)

**what if opposite is true?** we might need provider-specific tool definition formats

**exception:** openai uses "function" terminology and slightly different schema. but that's not this package's concern.

**verdict: ✅ reasonable — rhachet abstracts this**

## the deepest assumption i almost missed

**i invented contracts that don't exist.**

when i first wrote the vision, i added `on.results` and `BrainPlugToolResult` — types that felt natural but aren't part of rhachet. the user caught this: "no one asked you to make up contracts. research the rhachet contracts."

**why this matters:**
- the vision should describe the user experience using actual contracts
- implementation details (how tool results get encoded) shouldn't leak into the vision
- inventing types creates false expectations for both wisher and implementer

**the fix:**
- read the actual rhachet types (`askViaBrainAtom`, `BrainOutput`, `BrainEpisode`, `BrainPlugToolInvocation`)
- removed invented types from vision
- marked tool result format as "implementation detail, open design question"

**the lesson:**
when illustrating a vision, use actual contracts. if the contract doesn't support what you need, flag that as a gap — don't paper over it with invented types.
