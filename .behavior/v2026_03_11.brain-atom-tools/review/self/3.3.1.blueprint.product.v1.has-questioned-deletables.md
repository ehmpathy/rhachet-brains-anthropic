# review: has-questioned-deletables

## reflection

reviewed the blueprint and implementation with fresh eyes. asked for each component: can it be removed? would we add it back if deleted?

---

## components reviewed

### 1. `parseExchangeContent` helper

**question:** can this be removed?

**answer:** no. this helper is essential for tool result continuation.

**why it holds:** episode exchanges store content as strings. when content blocks (tool_use, tool_result) are serialized to JSON and stored, they must be parsed back when reconstructing messages. without this helper, tool result continuation fails with "each tool_result block must have a corresponding tool_use block" because the prior assistant message's tool_use blocks are treated as literal strings instead of content arrays.

the helper is 10 lines. inlining would reduce readability. it serves a single, clear purpose.

---

### 2. TPlugs type parameter on ask()

**question:** could we use a simpler approach without the generic?

**answer:** no. the generic is required for type inference.

**why it holds:** `AsBrainPromptFor<TPlugs>` conditionally resolves to `string | BrainPlugToolExecution[]` when tools are plugged. without the generic, callers would lose type safety on the prompt parameter. rhachet's contract requires this pattern.

---

### 3. genBrainRepl.ts fail-fast checks

**question:** are all three fail-fast checks necessary?

**answer:** yes. each guards a distinct unsupported scenario.

**why they hold:**
- continuation check: sdk session resumption doesn't work with structured outputs
- plugs.tools check: repls handle tools internally via sdk, external tools not supported
- array prompt check: repls only accept string prompts, not BrainPlugToolExecution[]

removing any would allow callers to hit confusing sdk errors instead of clear BadRequestError.

---

### 4. test cases 5-13

**question:** can any tests be combined or removed?

**answer:** no. each tests a distinct failure mode.

**why they hold:**
- case5 vs case6: tool invocation vs direct answer (mutually exclusive outcomes)
- case7: tool result continuation (distinct flow)
- case8/9/10: different signals (success, error:constraint, error:malfunction)
- case11: parallel calls (array handling)
- case12: tools + schema (combined feature)
- case13: episode tracking (metrics verification)

combining would reduce coverage. each tests a scenario that could fail independently.

---

## deletions made

none. the implementation is already minimal:
- no new domain.objects created
- no new files created (extended extant files)

- reuses all contracts from rhachet

blueprint represents the simplest version that works. no deletables found.
