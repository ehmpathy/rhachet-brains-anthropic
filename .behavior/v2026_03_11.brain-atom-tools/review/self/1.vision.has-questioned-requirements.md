## requirement 5: test "w/ and w/o output schema"

**who said this?** the wisher explicitly: "w/ and w/o output schema"

**this needs questioning:**

**current state:** atom.ask REQUIRES `schema: { output: z.Schema<TOutput> }`

**interpretation 1:** test tools WITH structured output AND WITHOUT structured output
- this means supporting `schema: undefined` when using tools
- return type becomes `string | TOutput` depending on schema presence
- significant contract change

**interpretation 2:** test that tool input schemas work correctly
- less likely — tool inputs always have schemas

**what if we didn't support "w/o output schema"?**
- tools only work with structured output
- simpler contract, but less flexible
- some use cases blocked (e.g., raw text generation with tool augmentation)

**question for wisher:**
> does "w/o output schema" mean:
> a) tools work when `schema: { output }` is NOT provided (raw text output)?
> b) something else?

**verdict: ⚠️ needs clarification**

## additional questions discovered

### Q1: continuation UX for tool results

the vision shows:
```ts
prompt: JSON.stringify({ tool_use_id: toolCall.exid, result: toolResult })
```

but anthropic expects proper `tool_result` content blocks in messages array, not stringified json in prompt.

**current proposal feels hacky.** should we:
- a) add `results` parameter for tool results? `{ on: { episode, results: [...] } }`
- b) use different message format internally?
- c) keep as-is and document the format?

**verdict: ⚠️ needs design decision**

### Q2: structured output + tools compatibility

flagged as open question in vision. this is **critical**.

if anthropic's `output_format` (beta) doesn't work alongside `tools`, we have a fundamental conflict.

**should validate before implementation.**

**verdict: ⚠️ must validate via spike**

### Q3: haiku support

haiku already doesn't support continuation with structured outputs. does haiku support tools at all? with structured output?

**verdict: ⚠️ needs testing**
