## new mechanisms reviewed

### 1. parseExchangeContent helper

**location:** `genBrainAtom.ts:110-122`

**code:**
```ts
const parseExchangeContent = (
  content: string,
): Anthropic.MessageParam['content'] => {
  if (content.startsWith('[')) {
    try {
      return JSON.parse(content) as Anthropic.MessageParam['content'];
    } catch {
      return content;
    }
  }
  return content;
};
```

**question:** does this duplicate extant functionality?

**search:** grep for `JSON.parse` in src/ found:
- `config.dao.ts:61` — parses JSON settings files
- `genBrainRepl.ts:168` — parses structured output
- `genBrainAtom.ts:176` — parses structured output

**analysis:** the extant JSON.parse usages serve different purposes:
- config files have known schemas
- structured output parses zod-validated JSON
- parseExchangeContent handles anthropic content blocks with unknown shape

no extant utility handles anthropic content block parse. this is a new concern specific to tool use continuation.

**verdict:** not duplicate. new domain concern.

## issue found

### redundant ternary in tool_result serialization

**location:** `genBrainAtom.ts:101-103`

**code:**
```ts
content: JSON.stringify(
  exec.signal === 'success' ? exec.output : exec.output,
),
```

**issue:** both branches return `exec.output`. the ternary is dead code.

**analysis:** likely meant to handle error cases differently. but `exec.output` works for all signals since:
- success: `TOutput`
- error:constraint/malfunction: `{ error: Error }`

JSON.stringify handles both correctly.

**verdict:** cosmetic issue. the ternary should be simplified to just `exec.output`. no behavioral impact.

**action:** flag for cleanup. not blocking.
