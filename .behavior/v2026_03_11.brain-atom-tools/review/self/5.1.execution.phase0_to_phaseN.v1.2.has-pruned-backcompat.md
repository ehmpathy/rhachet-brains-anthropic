## analysis

### signature changes examined

| change | compatible? | why it holds |
|--------|-------------|--------------|
| `plugs?: TPlugs` | ✅ yes | optional — old callers omit it |
| `prompt: AsBrainPromptFor<TPlugs>` | ✅ yes | union includes `string` when no TPlugs |
| `calls` in output | ✅ yes | additive field — old callers ignore it |
| `output` can be null | ⚠️ change | only when tools invoked — old callers never trigger this |

### why no backcompat shims exist

**the design is additive**

callers who don't pass `plugs.tools`:
- `prompt` accepts `string` (same as before)
- `output` is always defined (same as before)
- `calls` is `null` (can be ignored)

callers who pass `plugs.tools`:
- new behavior — no prior contract to maintain

**no shims were needed because**:
1. TypeScript generics handle the type variance
2. Optional parameters preserve old call sites
3. The "null output" case only activates with tool use

### backcompat code searched for

checked for:
- ❌ deprecated parameters — none found
- ❌ version checks — none found
- ❌ legacy code paths — none found
- ❌ re-exports for old names — none found
- ❌ "assume default if absent" logic — none found

### the one real change

`output` can be `null` when model defers to tools.

this is intentional per the blueprint:
> "output and calls are independent"

callers must understand the tool workflow:
`ask → calls → execute → ask again → output`

this is not a backcompat concern — it's the new contract.

## verdict

✅ no backcompat shims needed or added

the implementation is additive by design.
old callers continue to work unchanged.
new callers get tool support.

no backwards compatibility was requested in the wish,
and none was artificially added "to be safe".
